AI/Chat Architecture

Technical

This is the feature that makes Fantasy League Hub special: an AI assistant that actually understands your league's full history. "Who had the best draft in 2023?" "What's my all-time record against Mike?" "Generate a recap of last season." The challenge is context — a league with 10 years of history across 12 managers has thousands of matchups, transactions, and draft picks. You can't dump all of that into a single LLM prompt. The architecture needs to figure out what data is relevant to each question, retrieve it efficiently, and give the LLM enough context to answer accurately without hallucinating stats.

RAG + Tool Use (pgvector)

1 User asks: "Best draft in 2023?"
↓
2 LLM decides: call get_draft_results(2023)
↓
3 Tool queries Postgres for 2023 draft data
↓
4 pgvector finds similar past discussions
↓
5 LLM receives structured data + context
↓
6 Generates answer with real stats cited
Combine two powerful patterns: RAG (Retrieval-Augmented Generation) using pgvector embeddings in your existing Postgres database, and LLM tool use where the AI can call specific functions to query your data. In step 2, the LLM decides which tools to call — get_draft_results(), get_head_to_head(), get_standings() — based on the question. These tools run real Postgres queries and return structured data. pgvector (step 4) adds semantic search for fuzzier queries. The LLM gets real data, not hallucinated stats. I recommend this because tool use gives you precision (exact stats from your database) while RAG gives you flexibility (fuzzy/narrative queries), and pgvector lives in Postgres so there's no extra infrastructure.

Works well when

  • Most accurate — LLM gets real data from your database via tools
  • pgvector in Postgres — no separate vector database needed
  • Handles both precise stats and narrative/fuzzy queries

Watch out for

  • Tool definitions need maintenance as your data model evolves
  • Multi-step tool calls add latency (2-4 seconds per query)

After each data sync, pre-compute condensed text summaries of your league data: a league overview doc (champions, records, rivalries), per-season summaries, and per-manager profiles. When a user asks a question, stuff the relevant pre-computed summaries into the system prompt — no tool calls needed, no vector search. The LLM just reads the summaries and answers. This is simpler to build and faster at response time (no tool call round-trips). The trade-off is that the summaries may not contain the specific detail the user is asking about, and updating them after every sync costs LLM tokens.

Works well when

  • Simplest architecture — no tools, no vector DB, just context stuffing
  • Fastest response time — no round-trips for data retrieval
  • Pre-computed summaries can power non-chat features too (recaps)

Watch out for

  • Summaries may miss the specific detail a user asks about
  • Generating summaries after every sync costs LLM tokens
  • Harder to answer precise stat queries ("exact score in Week 7 2021")

Option C
Chosen

Pure Tool Use (SQL Agent)

1 User: "Longest win streak ever?"
↓
2 LLM generates SQL query
↓
3 Execute against Postgres (read-only)
↓
4 Return query results to LLM
↓
5 LLM formats into natural language
Give the LLM your database schema and let it write SQL queries directly. In step 2, the AI generates a SQL query based on the user's question and your schema. The query runs against a read-only Postgres replica, and the results come back as data the LLM formats into a natural language answer. This is the most flexible approach — the AI can answer any question that's answerable from your data. No pre-built tools, no embeddings, no summaries. The risk is that LLM-generated SQL can be wrong, slow, or occasionally dangerous (even with read-only access, a bad query can overload the DB).

Works well when

  • Most flexible — can answer any question from the data
  • No tool definitions to maintain — schema is the interface
  • Can handle complex, multi-join analytical queries

Watch out for

  • LLM-generated SQL can be wrong or inefficient
  • Bad queries can overload the database
  • Needs guardrails — query timeout, row limits, read-only access

Build all three retrieval strategies (tool use, pre-computed summaries, vector search) and add a routing layer that picks the best one for each question. "What was my exact score in Week 3?" routes to tool use. "Give me a recap of last season" routes to pre-computed summaries. "Who reminds you of a 2019-era manager?" routes to semantic search. A narrative engine also runs after each season/week to generate storylines, rivalry updates, and power rankings automatically. This is the most capable approach but also the most complex — it's what you'd build for a polished v2 or v3, not v1.

Works well when

  • Best possible answer quality — right strategy for each question
  • Automated narratives (recaps, storylines) are a killer feature
  • Can evolve each strategy independently

Watch out for

  • Most complex — three retrieval systems + a router
  • Router itself needs to be accurate or answers degrade
  • Significantly more LLM cost (routing + retrieval + generation)

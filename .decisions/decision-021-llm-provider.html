LLM Provider & SDK

Technical

Decision 006 locked in "RAG + Tool Use" as the AI architecture, but didn't specify which LLM to use. This choice affects everything: how tools are defined (each provider has its own format), how responses stream to the iOS client, which embedding model to use for pgvector, and your ongoing API cost per chat message. The right answer depends on context window size needed, tool calling reliability, and whether you want to stay in one provider's ecosystem.

Use Claude Sonnet (currently claude-sonnet-4-6) via the official Anthropic TypeScript SDK. Claude has best-in-class tool use reliability — it correctly decides when to call tools and chains multi-step tool calls well. The 200K context window means you can stuff a lot of league history directly into the prompt as a fallback before reaching for pgvector. Embeddings use voyage-3 (Anthropic's recommended partner) or OpenAI's embedding model separately.

Works well when

  • Tool calling reliability is the top priority
  • 200K context window reduces how much you need pgvector
  • You're already building with Claude Code — same ecosystem

Watch out for

  • Embeddings need a separate model (Voyage or OpenAI)
  • Slightly higher cost vs. GPT-4o-mini at scale

Use GPT-4o or GPT-4o-mini via the official OpenAI TypeScript SDK. The huge advantage: text-embedding-3-small gives you embeddings from the same provider at very low cost ($0.02/M tokens), which simplifies the pgvector pipeline — one API key covers both chat and embeddings. GPT-4o-mini is significantly cheaper for high-volume requests. Function calling is mature and well-documented.

Works well when

  • One API key for both chat and embeddings is simpler
  • Cost is a concern — 4o-mini is much cheaper at scale
  • Largest community, most examples for RAG + tool use patterns

Watch out for

  • 128K context vs Claude's 200K — matters for large league histories
  • Tool calling slightly less reliable than Claude on complex multi-step tasks

Use the Vercel AI SDK — a unified TypeScript library that abstracts over Anthropic, OpenAI, Google Gemini, and others behind a single interface. You write tool definitions once and can swap the underlying model by changing one line. Streaming, tool use, and structured output work the same way regardless of provider. The tradeoff is a layer of abstraction that can lag behind provider-specific features.

Works well when

  • You want to swap providers later without rewriting tool definitions
  • Portability and avoiding vendor lock-in matters
  • Good streaming utilities built in for Node.js

Watch out for

  • Abstraction layer can lag on new provider features
  • Additional dependency; less control over request shape
  • Overkill for a single-provider app

Use Claude for chat and tool calling (best reliability), but use OpenAI's text-embedding-3-small for generating pgvector embeddings (cheapest at scale: $0.02/M tokens). Two API keys, but each is used for what it's best at. This is the "best of both" approach — the embedding model choice doesn't affect chat quality, and OpenAI embeddings are the industry standard for RAG pipelines.

Works well when

  • You want Claude's tool calling with the cheapest embeddings
  • Embedding volume is high (all historical matchups, chats, etc.)
  • Best-of-breed over simplicity

Watch out for

  • Two API keys and two billing accounts to manage
  • Slightly more complex setup than a single-provider approach

LangChain.js wraps the entire RAG pipeline — document loaders, text splitters, vector store retrievers, rerankers, and chain abstractions — so you assemble components rather than write plumbing. It supports all major LLM providers. The tradeoff: LangChain adds significant abstraction layers that obscure what's actually happening in your RAG pipeline, making debugging harder. Its value is in multi-document pipelines with complex chains — which this app doesn't need. For a focused tool-use system with known Prisma queries, direct SDK calls are more transparent.

Works well when

  • Many document types and complex chain logic (not this app's case)
  • Team already knows LangChain
  • Pre-built retriever abstractions save time on pgvector setup

Watch out for

  • Heavy abstraction makes debugging and observability harder
  • Frequent breaking changes between LangChain versions
  • Overkill for a focused tool-use system — adds complexity without benefit

Use Claude on the backend (same as Option A), but add swift-ai-sdk on the iOS client to handle streaming responses with type-safe Codable structs. The Swift library is a port of Vercel AI SDK with support for all major providers. Important caveat: this is a single-developer open-source project at an early stage — the Jahnel Emerging (R4) caution applies directly. The safer iOS approach is URLSession + AsyncStream with a hand-rolled SSE parser, which is ~50 lines and has no external dependency.

Works well when

  • You want provider-agnostic streaming on iOS (swap Claude ↔ GPT)
  • Type-safe Codable structured outputs matter for rich response cards

Watch out for

  • Single-developer project — risk of abandonment or breaking changes
  • Tool execution still goes through the Express backend — not a standalone solution
  • URLSession + AsyncStream handles SSE in ~50 lines without a dependency

Player Detail Layout

Interaction

When a user taps a player name anywhere in the feed, they land on a Player Detail view. This view needs to compose four distinct data types — current rankings, weekly stats, social sentiment (Reddit/Bluesky mentions), and recommendation signals — into a single coherent screen. The challenge is that these data types have different shapes and different urgency levels. This decision is about which one leads, how the others are presented, and whether the layout uses scrolling sections, tabs, or a hybrid.

A sticky stats hero at the top (ranking, projected points, season average, last-3-week trend) stays visible as the user scrolls down into a chronological signal feed below. Stats answer the "what do I know about this player" question immediately. The feed below surfaces recent social mentions, ranking changes, and recommendation signals in time order. This maps naturally to how fantasy users think: check the number first, then understand the narrative. I recommend this because the data hierarchy matches user intent — stats are the fact, signals are the context.

Works well when

  • Users come for a quick stats check — answer that immediately above the fold
  • Signal feed grows naturally as more sources come online over time
  • Sticky hero keeps key numbers visible while scrolling through signals

Watch out for

  • Hero takes vertical space — signal feed starts lower on smaller phones
  • Mixed signal types in one feed can feel unstructured without clear grouping

The signal feed starts immediately below a compact two-line header. Stats are collapsed into a drawer — tap "Stats" to expand a panel showing projections, averages, and rank history. This prioritizes recency: if you tapped a player from the feed, you already know why. You want to read more about the signal that caught your eye, not see a stat table you already know. Stats are one tap away but don't push the feed down. The tradeoff is that the drawer interaction adds a step for users who arrive cold.

Works well when

  • Entry point is always a specific signal — users want more context on what they just saw
  • Signal feed starts above the fold on all screen sizes
  • Most users already know the player's baseline stats

Watch out for

  • Drawer adds a tap for users who check stats first — against established pattern
  • Collapsed stats can feel like the app is hiding useful info

Three tabs — Stats, Signals, Rankings — each with its own full-height scrollable section. Stats gets a stat grid, Signals gets the social/sentiment feed, Rankings gets the ECR history chart. Users navigate between sections with a tab bar below the player header. This is the clearest information architecture: each data type gets undivided space. The tradeoff is that users can only see one dimension at a time and have to tab between them to build a complete picture of the player.

Works well when

  • Each data type needs full vertical space to breathe (especially charts)
  • Users have a clear mental model — "I want to check rankings" → Rankings tab
  • Each section grows independently as more data types are added

Watch out for

  • Tab switching friction — users have to know to look in other tabs
  • Signals and ranking moves feel disconnected from the stats that contextualize them

A compact scorecard header summarizes the player's status — projected points, ECR rank, signal count, trend direction — in a single glance. Below it, three expandable section rows (Stats, Social Signals, Rankings) expand inline when tapped, revealing the full content without navigating away. This gives the most information density: the header answers "should I care about this player right now" in one line, and sections expand only for what the user wants to dig into. The downside is that all sections start collapsed, requiring taps to see any content.

Works well when

  • Users want a quick yes/no on a player before deciding to dig in
  • Scorecard summary reduces need to open any section for casual checks
  • Expandable pattern is familiar from iOS Settings and similar apps

Watch out for

  • All content hidden behind taps — users who want to read signals see nothing by default
  • Scorecard summary logic is complex to compute correctly across all data types

System Prompt & League Context Injection

Technical

Before GPT-4o can answer "who's been my most consistent manager?", it needs to know who you are, which league you're asking about, how scoring works, and who the managers are. How that context gets into the model — and how much of it — determines answer quality, API cost, and how well the model grounds its answers in your actual league data versus hallucinating generic fantasy advice.

Lean Header + Tools + RAG

SYSTEM: You are a fantasy football analyst for the BIG12 League (Sleeper, PPR scoring). You're talking to Frank Nardone (team: "Frank's Squad"). Managers: Frank, Jake, Tyler, Sarah... (12 total) Seasons on record: 2019–2024 // ~200 tokens — identity, names, nothing more TOOLS AVAILABLE: - getStandings(season) - getMatchups(season, week?) - getDraftResults(season) - getManagerStats(managerId, season?) RAG RETRIEVED (this query): "2024 playoff results: Frank beat Jake..." // only injected when semantically relevant
The system prompt stays lean — just enough for the model to know who's who (~200 tokens). When a question needs specific stats, the model calls tools (hitting your V1 endpoints). When a question is about narrative or history ("tell me about our rivalry"), RAG injects relevant text chunks from the embedding table. This three-layer approach keeps context costs low on simple queries and only pays for heavy context when the question actually needs it.

Works well when

  • Balanced cost — simple questions are cheap, complex ones are thorough
  • Tools guarantee accuracy for stats (no hallucination)
  • RAG handles narrative and historical context efficiently

Watch out for

  • Requires tool calls to be working before chat quality is high
  • RAG relevance tuning needed to avoid injecting wrong context

Serialize all league analytics — standings history, every manager's record, draft grades, matchup results — and inject it into the system prompt on every request. No tools needed, no RAG needed. The model has everything and can answer anything without making additional calls. Sounds great, but a 6-season league with 12 managers produces 8,000–15,000 tokens of data. That's $0.08–0.15 per question just in input tokens, and it fills most of GPT-4o's context window leaving little room for conversation history.

Works well when

  • No tool call latency — model has everything upfront
  • Simplest architecture — just one big prompt, no moving parts

Watch out for

  • Very expensive — $0.08–0.15 per message in input tokens alone
  • Fills context window — limits conversation history and reply length
  • Doesn't scale as league history grows

Option C

RAG-Only Context

SYSTEM: You are a fantasy football analyst. Use only the context provided below to answer. // generic — no league identity injected RAG RETRIEVED (top 5 chunks): "2024 Week 7: Frank 142 vs Jake 98..." "2023 Draft: Frank picked Tyreek Hill..." "2024 standings: Frank 9-5, Jake 8-6..." "2022 championship: Frank def. Tyler..." "Manager history: Frank joined 2019..." // hope the right 5 chunks cover the question
Skip the league header entirely — let the vector search do all the work. When a user asks a question, embed it, find the top-5 most semantically similar chunks from the embedding table, and inject only those. No tools. The system prompt is generic. This is clean and cheap on simple questions, but falls apart on structured stat queries ("what was my best scoring week?") where the right answer requires database aggregation, not text retrieval. RAG alone can't reliably answer questions that need real-time computation.

Works well when

  • Questions are narrative/historical ("tell me about season 3")
  • Very low cost per query

Watch out for

  • Poor on structured stat queries — wrong chunks get retrieved
  • Model doesn't know who it's talking to without the league header
  • Hallucination risk when the right chunk isn't in the top-5

Tool-First, No RAG

SYSTEM: You are a fantasy analyst for the BIG12 League. You have these tools: - getStandings(season) - getMatchups(season, week?, managerId?) - getDraftResults(season) - getManagerStats(managerId, season?) - searchLeagueHistory(query) ← does text search // No RAG injection — model calls tools // for EVERYTHING, including narrative
Forget RAG entirely — give the model a richer set of tools including a searchLeagueHistory(query) tool that runs a database text search when needed. The model decides what to call. This keeps the system prompt lean and makes tool calls the single source of truth. It works well for structured stat questions but creates extra round-trips for every question (even simple ones) and requires a robust search tool to handle anything narrative or historical that RAG would normally cover.

Works well when

  • All answers come from the DB — no stale embedding index to maintain
  • Single source of truth — tools always return current data

Watch out for

  • Every question requires at least one tool call — adds latency
  • Narrative/historical queries need a capable search tool to substitute for RAG

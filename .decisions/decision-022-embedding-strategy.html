Embedding Strategy & pgvector Schema

Technical

Per the Jahnel Group principles, chunking strategy is a first-class architectural decision — not an implementation detail to figure out later. This decision defines what data gets embedded, how it's chunked, where the vectors live in Postgres, and when embeddings are generated. The choice shapes the Prisma migration, the BullMQ embedding job, and ultimately how good the AI's retrieval is when answering questions about your league.

One dedicated embeddings table holds all vectors with a polymorphic reference back to the source: (sourceType, sourceId) points to a Matchup, Season, Manager, etc. The content field stores the human-readable text that was embedded, so you can show it when debugging retrieval. Each chunk is an independent row — a manager's career might produce 3 chunks, a full season summary produces 1. Clean separation: the main schema is untouched, and you can re-embed everything by truncating one table.

Works well when

  • Clean separation — main schema untouched, easy to re-embed
  • Supports multiple chunk sizes per source without schema changes
  • Debug retrieval by reading content field directly
  • Easy to wipe and rebuild: TRUNCATE embeddings

Watch out for

  • JOIN required to get source data alongside embedding results
  • No FK enforcement — sourceId is a string, not a typed FK

Add a nullable embedding vector(1536) column directly to each model that needs one: Season, Manager, Matchup. No new table — vectors live alongside the data they describe. The big limitation: one row = one embedding. A manager with 14 seasons of history produces one vector, which forces you to cram everything into one chunk or lose detail. Works if queries are always at the model level ("tell me about manager X") but breaks for fine-grained retrieval ("what happened in week 7 of 2019").

Works well when

  • Simpler schema — no new table, no polymorphic joins
  • One-to-one: each row has exactly one embedding
  • Strong FK enforcement — vector is part of the model

Watch out for

  • One chunk per row forces oversized chunks — hurts retrieval quality
  • Can't have multiple chunk sizes per source
  • Re-embedding requires ALTERing production columns

Season Summaries Only (Skip pgvector)

Only high-level narrative summaries are embedded. Individual matchups/picks accessed via tools directly.
Skip embedding individual matchups, managers, and picks entirely. Instead generate high-level narrative summaries per season (e.g. "2022: franknardone won the championship with a 10-3 record, beating blake in the finals 147-112…") and embed those. All specific data still comes from structured tool calls — pgvector is only used for broad narrative context. Fastest to implement and lowest embedding cost, but retrieval is coarse.

Works well when

  • Fastest to ship — minimal schema, minimal embedding cost
  • GPT-4o tool calls handle all specific stats anyway
  • Good enough for most "what happened in 2022" questions

Watch out for

  • Poor retrieval for specific queries ("who did I beat most in week 10")
  • Narrative summaries may go stale as data changes
  • Harder to retrofit fine-grained embeddings later

Don't add pgvector at all for the initial AI chat release. Build the entire chat experience using structured tool calls only (query_standings, query_h2h, query_scoring, etc.). The tools cover 90% of questions users will ask. Add pgvector as a follow-on once you've seen real usage and know which query types the tools can't handle. This respects the "ship real systems and learn from what breaks" principle from the Jahnel expert tier.

Works well when

  • Fastest path to shipping AI chat — no embedding pipeline to build first
  • Tools handle the structured stats queries that 90% of users will ask
  • Lets real usage patterns inform what to embed

Watch out for

  • Fuzzy/narrative questions ("who's my biggest rival historically") fall back to LLM guessing
  • Retrofitting pgvector after launch is a second migration and deployment

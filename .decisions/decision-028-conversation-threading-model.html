Decision 28 of 28

Conversation Threading Model

Interaction

If a user chats with the AI about their BIG12 league on Monday, then opens the app on Thursday, what do they see? A blank slate? Their prior conversation still open? A list of past chats to pick from? This shapes the entire navigation design for the Chat tab and determines how conversation history storage (decision 025) actually surfaces to the user.

One ongoing conversation per league, automatically scoped. Opening the Chat tab shows your leagues as entry points — tapping BIG12 drops you into the thread where you left off. The model always knows which league you're talking about because it's baked into the thread. No thread management needed from the user. If you have 3 leagues, you have 3 independent conversations. Simple, matches how users think about their leagues.

Works well when

  • Natural mapping: one league = one conversation
  • No thread UI complexity — zero decisions for the user
  • League context is automatic — never ambiguous

Watch out for

  • Very long threads over time — need a turn window to keep context manageable
  • Can't ask a fresh question without old context polluting it

Same as Option A — one primary thread per league — but with a "New Chat" button that archives the current thread and starts fresh. The archived threads are visible but not surfaced prominently. This solves the "long thread getting stale" problem and lets users start a focused conversation without old context dragging along. It's a small addition to Option A with meaningful UX benefit — especially when the user wants to ask something completely unrelated to their last conversation.

Works well when

  • Combines continuity (default) with the escape valve of a fresh start
  • Keeps context clean without forcing users to manage threads
  • Minimal added complexity over Option A

Watch out for

  • Slightly more complex DB model — threads need a status (active/archived)
  • Archived thread list could grow messy over time

Ephemeral Sessions

AI Chat — BIG12 League
Conversation resets each session
What would you like to know about your BIG12 league?
Who's my biggest rival?
Based on your record, Jake has been your closest matchup...
chat history gone on next app open
The chat tab opens fresh every time. Within a session, the model remembers your conversation. Close the app and everything resets — next time you open it, it greets you as if you've never chatted before. No DB storage needed for messages (aligns with Option D from decision 025). Simplest to build, but feels low-quality compared to every chat product users interact with. The absence of history also removes any "wow, it remembered what we talked about" moment.

Works well when

  • Simplest implementation — no storage, no thread model
  • No stale context pollution across sessions

Watch out for

  • Every session starts cold — no continuity, feels disposable
  • Users re-ask the same questions every time they open the app

A full ChatGPT-style interface — users explicitly create threads, can name them, and switch between multiple active conversations about the same league. One thread for "draft analysis", another for "head-to-head history", another for "start/sit decisions". Most flexible, most powerful, but requires users to manage threads themselves. Adds a thread list view, thread creation UI, and thread naming — significant scope beyond the core chat experience. Worth it if this becomes a power feature, overkill for a V2 launch.

Works well when

  • Power users want to keep different conversations separate
  • Full ChatGPT-style familiarity for frequent users

Watch out for

  • Significant extra UI — thread list, creation flow, naming
  • Overcomplicates V2 launch — this is more of a V3 refinement

Player Resolution Strategy

Technical

Every ingestion job writes a raw player name string -- "Patrick Mahomes", "P. Mahomes", "Mahomes II", "Pat Mahomes" could all refer to the same person. You need a way to resolve these raw names to canonical Player records in your new Player table. Your build plan says "Levenshtein or similar" with ambiguous matches staying unresolved until LLM disambiguation in Phase 5. This decision is about the specifics: where does the canonical roster come from, how does fuzzy matching work, and how do you handle the ~5-10% of names that are genuinely ambiguous?

Alias Table + Exact-First

1 Ingest raw name from source
↓
2 Exact match against PlayerNameAlias table
↓
if no exact match ↓
3 Levenshtein against Player.fullName (threshold ≤ 2)
↓
if single match ↓
4a Auto-resolve + save alias for next time
if ambiguous or no match ↓
4b Leave unresolved (playerId = null)
The simplest approach that works. In step 2, check the PlayerNameAlias table first -- this is a lookup table of known name variants ("P. Mahomes" → Patrick Mahomes). If no alias matches, step 3 runs Levenshtein distance against all Player.fullName values with a strict threshold (≤ 2 edits). If exactly one player matches, auto-resolve and save the new alias so it's instant next time (step 4a). If multiple players match or none do, leave it unresolved (step 4b) for Phase 5 LLM disambiguation. The alias table is the key insight -- it's a self-building cache that makes the system faster over time. I recommend this because it's simple, deterministic, and the alias table handles 90%+ of cases after the first few runs.

Works well when

  • Most names are close to canonical ("P. Mahomes" → "Patrick Mahomes")
  • The alias table self-builds -- resolution improves automatically over time
  • No external dependencies, just string comparison in PostgreSQL or JS

Watch out for

  • Levenshtein is O(n*m) per comparison -- fine for ~2000 NFL players but be aware
  • Nicknames like "Hollywood Brown" won't match "Marquise Brown" (distance too high)

PostgreSQL Trigram

1 Ingest raw name from source
↓
2 Check alias table (exact match)
↓
3 pg_trgm similarity search (threshold ≥ 0.4)
↓
4 Rank by similarity score, take top if ≥ 0.6
↓
5 Auto-resolve or leave unresolved
Use PostgreSQL's built-in pg_trgm extension (already available since you're on PostgreSQL 17) for fuzzy matching instead of Levenshtein. Trigram similarity breaks names into 3-character chunks and compares overlap -- it handles typos, abbreviations, and reorderings better than pure edit distance. In step 3, run SELECT * FROM player WHERE similarity(full_name, 'P. Mahomes') > 0.4 ORDER BY similarity DESC. If the top result scores above 0.6, auto-resolve. The big advantage: pg_trgm supports GIN indexes, so fuzzy searches are fast even at scale.

Works well when

  • GIN index makes fuzzy search fast -- no scanning every player row
  • Handles abbreviations and typos better than Levenshtein
  • Built into PostgreSQL, no external library needed

Watch out for

  • Requires raw SQL -- Prisma doesn't support pg_trgm natively
  • Still can't resolve nicknames or slang ("Hollywood" → "Marquise Brown")

Roster-Seeded Lookup

1 Weekly: pull full NFL roster from SportsDataIO
↓
2 Upsert Player table with canonical names + IDs
↓
3 On ingest: exact match alias table
↓
4 Fallback: Levenshtein against Player.fullName
↓
5 Auto-resolve or leave unresolved
Same resolution logic as Option A, but the key difference is where the canonical Player roster comes from. Instead of manually seeding players, use SportsDataIO's roster endpoint (which you already have via the free tier from Decision 14) to automatically populate and update the Player table weekly. This means the Player table always has current NFL rosters with official names, teams, positions, and external IDs. Steps 3-5 are the same alias + Levenshtein flow -- but now you have a reliable, auto-updating roster to match against.

Works well when

  • Player table stays current automatically -- new signings, trades, rookies
  • SportsDataIO provides external IDs that link across platforms
  • Reuses a data source you're already paying for (free tier)

Watch out for

  • Uses some of your 1,000 monthly SportsDataIO API calls for roster syncs
  • Still need the same fuzzy matching logic on top -- roster source doesn't change that

Curated Dictionary

1 Maintain JSON dict: nicknames → canonical names
↓
2 On ingest: check dictionary first
↓
3 Fallback: exact + fuzzy match on Player table
↓
4 Log unresolved names for manual review
↓
5 Periodically add new mappings to dictionary
Start with a hand-curated JSON dictionary of known name variants and nicknames. "Hollywood" → "Marquise Brown", "CMC" → "Christian McCaffrey", "Lamar" → "Lamar Jackson" (when context is NFL). This handles the cases that no fuzzy matching algorithm can -- slang, nicknames, abbreviations that are culturally specific. The dictionary is checked first, then normal fuzzy matching runs as fallback. Unresolved names get logged for periodic manual review, and you grow the dictionary over time.

Works well when

  • Handles nicknames and slang that algorithms can't (CMC, Hollywood, etc.)
  • Deterministic -- same input always gives same output
  • Dictionary becomes a valuable asset, especially for Reddit/social content

Watch out for

  • Requires manual curation effort -- someone has to maintain the dictionary
  • New players or trending nicknames won't resolve until added
  • Ambiguous single names ("Lamar") could match wrong sport/context

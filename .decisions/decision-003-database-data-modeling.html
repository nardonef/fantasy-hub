Decision 3 of 12

Database & Data Modeling

Technical

This is arguably the most important technical decision for Fantasy League Hub. You're aggregating data from three different providers (Yahoo, ESPN, Sleeper) that each structure their data differently — different IDs, different season formats, different ways of representing matchups, rosters, and transactions. The database needs to normalize all of this into a unified model that supports fast analytics queries across years of history. The choice here affects query performance, how easy it is to add new providers later, and how the AI assistant retrieves league context.

PostgreSQL with Prisma ORM — the most popular combo in the Node.js ecosystem. Prisma gives you a type-safe query builder that auto-generates TypeScript types from your schema, so your Express API gets compile-time safety on every database query. PostgreSQL is rock-solid for relational data and excels at the complex joins you'll need (e.g., "give me all matchups between Manager A and Manager B across all seasons and providers"). You'd use pgvector (a Postgres extension) for the AI feature — storing embeddings of league events so the chat assistant can do semantic search. I recommend this because the relational model is the natural fit for league data, Prisma keeps your queries type-safe, and pgvector means you don't need a separate vector database for AI.

Works well when

  • Type-safe queries — Prisma catches errors at compile time
  • Complex joins are fast (head-to-head, cross-season analytics)
  • pgvector for AI embeddings — no separate vector DB needed

Watch out for

  • Schema migrations can be tricky as the model evolves
  • Prisma adds a query engine layer — slight overhead vs raw SQL

Query Example
db.select()

Same PostgreSQL database, but with Drizzle ORM instead of Prisma. Drizzle is newer and takes a different philosophy: your schema is defined in TypeScript code (not a separate schema file), and queries look almost like SQL — so if you know SQL, you already know Drizzle. It's thinner than Prisma (no query engine binary), generates SQL you can read and predict, and runs faster for complex queries. The trade-off is a smaller ecosystem and slightly more manual setup for things Prisma auto-generates (like relations). Still gets pgvector for AI embeddings.

Works well when

  • Thinner abstraction — queries map directly to SQL
  • Better performance for complex analytics queries
  • Schema-as-TypeScript-code feels natural

Watch out for

  • Newer — smaller ecosystem and community
  • Relations require more manual setup than Prisma

Use both: PostgreSQL for your normalized, structured data (users, leagues, matchups, standings) and MongoDB for the raw, messy stuff — provider API response snapshots, transaction logs, draft event streams, and chat history. The idea is that Yahoo, ESPN, and Sleeper all return different JSON shapes, so you store the raw responses in MongoDB as-is (easy to debug, reprocess, or re-sync) while the normalized, query-friendly version lives in Postgres. It's a solid pattern when you're ingesting data from external APIs with unpredictable schemas.

Works well when

  • Raw provider data is messy and varies — MongoDB handles it naturally
  • Easy to reprocess raw data if your normalized model changes
  • Each database does what it's best at

Watch out for

  • Two databases to manage, backup, and keep consistent
  • Queries that span both stores are complex
  • More infrastructure cost and operational overhead

PostgreSQL + JSONB Flex Columns

matchups (relational + flexible)
id uuid PK
league_id uuid FK
week, team_a, team_b standard cols
score_a, score_b decimal
provider_data jsonb
computed_stats jsonb
↓ JSONB is indexed & queryable
Query provider-specific fields
WHERE provider_data->>
  'projection' > '100'
One PostgreSQL database, but use JSONB columns alongside traditional relational columns for the parts that vary between providers. Core fields (scores, weeks, team IDs) are normal columns you can index and join on. Provider-specific extras (Yahoo's projected points format, ESPN's specific stat categories, Sleeper's player metadata) go into a JSONB column — fully indexed, fully queryable, but flexible enough to store different shapes without schema changes. This gives you the Option C benefit (handling messy provider data) without the Option C cost (two databases). It's the pragmatic middle ground.

Works well when

  • One database — simpler ops, backups, and consistency
  • JSONB handles provider-specific data without schema changes
  • Still fully queryable and indexable via GIN indexes

Watch out for

  • JSONB queries are slower than native columns for heavy analytics
  • No compile-time type safety on JSONB fields
  • Can become a junk drawer if not disciplined

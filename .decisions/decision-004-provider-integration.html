Decision 4 of 12

Provider Integration Strategy

Technical

The heart of Fantasy League Hub is pulling data from Yahoo, ESPN, and Sleeper and making it feel like one unified experience. Each provider is different: Yahoo has an official OAuth API, ESPN has an unofficial API you authenticate via cookies, and Sleeper has a public read-only API with no auth needed. The question is how you architect the sync — do you build custom adapters for each, use an abstraction layer, batch everything in background jobs, or take a more event-driven approach? This decision affects how reliable syncing is, how fast new providers can be added, and how fresh the data stays.

Build a typed adapter (a translation layer) for each provider that converts their specific data format into your unified model. Each adapter implements the same interface — fetchLeagues(), fetchMatchups(), fetchRoster(), etc. — so the rest of your app doesn't care which provider the data came from. BullMQ (which we already have from our backend choice) schedules and runs sync jobs: full historical imports when a league is first connected, then incremental updates on a schedule. Each job is retryable, has error handling, and reports progress. I recommend this because it's the most straightforward architecture that cleanly separates "how to talk to Yahoo" from "how to store a matchup," and BullMQ gives you reliable job processing with retry logic built in.

Works well when

  • Clean separation — adding a new provider means writing one adapter
  • BullMQ handles retries, rate limiting, and progress tracking
  • Easy to test — mock the adapter, test the sync logic

Watch out for

  • Full historical import can be slow (many API calls per season)
  • Each adapter needs ongoing maintenance as provider APIs change

Same adapter pattern as Option A, but add a live polling layer that checks providers frequently during game days (every few minutes) and less often off-season. When changes are detected — a trade happened, scores updated, waivers cleared — it emits events that trigger targeted sync jobs rather than full re-syncs. This enables near-real-time features: push notifications when a trade goes through, live score updates during games, waiver results as they process. More complex to build but unlocks the "live" features that make fantasy apps sticky.

Works well when

  • Near-real-time updates — scores, trades, waivers as they happen
  • Event-driven sync is more efficient than full re-imports
  • Enables push notifications and live dashboards

Watch out for

  • Significantly more complex — event bus, polling, change detection
  • Provider rate limits become a real concern with frequent polling
  • ESPN's unofficial API makes reliable polling fragile

ETL Pipeline (Extract-Transform-Load)

1 Extract: Pull raw JSON from providers
↓
2 Store raw data in staging tables
↓
3 Transform: Normalize to unified model
↓
4 Validate: Check data consistency
↓
5 Load: Upsert into production tables
↓
6 Compute: Refresh analytics/aggregates
Treat provider syncing as a formal data pipeline with distinct Extract, Transform, and Load phases. First, pull raw JSON from each provider and store it as-is in staging tables (step 2 — this is your safety net if the transform logic changes). Then a transform step normalizes into your unified model (step 3), validates consistency (step 4), and loads into production tables (step 5). Finally, a compute step refreshes analytics aggregates. This is the most robust approach for data integrity — you can reprocess any step independently, and the raw data is always preserved. It's what data engineering teams use for exactly this kind of multi-source aggregation problem.

Works well when

  • Most robust data integrity — raw data preserved, reprocessable
  • Transform logic can evolve without re-fetching from providers
  • Clear debugging — see exactly where data went wrong

Watch out for

  • More infrastructure — staging tables, pipeline orchestration
  • Higher latency — data goes through multiple stages
  • Over-engineered for v1 if you just need basic sync working

Skip the background sync entirely for v1. Instead, fetch data from providers on-demand when the user requests it, normalize it on the fly, cache it in Redis (short-term) and PostgreSQL (long-term). When someone opens their league dashboard, the API checks if we have fresh data — if not, it fetches from the provider right then. Historical data gets cached permanently after the first fetch. This is the simplest approach and gets you to a working product fastest. The downside is that first loads are slow (waiting for provider API calls) and you can't do things like push notifications since you're not actively monitoring for changes.

Works well when

  • Simplest to build — no background jobs, no polling, no queues
  • Fastest path to a working MVP
  • No wasted API calls — only fetch what users actually look at

Watch out for

  • Slow first loads — user waits for provider API calls
  • No background monitoring — can't push notifications on trades/scores
  • Provider rate limits can cause request failures during peak times

Decision 2 of 12

Backend & API Architecture

Technical

Fantasy League Hub's backend has to do a lot of heavy lifting: sync data from three different fantasy providers (Yahoo, ESPN, Sleeper), compute analytics across years of league history, serve an AI chat that understands your league context, and handle multi-user access with real-time updates. Since we chose SwiftUI Native for the frontend, the backend language doesn't need to match — we're free to pick whatever is best for the job. The key question is how much infrastructure you want to manage yourself versus how much you want a platform to handle.

A Node.js server with Express, written in TypeScript. This is the most battle-tested approach for building REST APIs. You get a massive ecosystem of libraries — BullMQ for background job queues (provider syncing), Passport for auth, and great LLM SDK support (Anthropic, OpenAI). Background workers handle the heavy provider sync jobs separately from API requests. Deploy to Railway or Fly.io with minimal DevOps. I recommend this because it's the most straightforward path to a production backend with all the pieces Fantasy League Hub needs — and the ecosystem for API integrations, job queues, and AI SDKs is unmatched.

Works well when

  • Largest ecosystem for API integrations and LLM SDKs
  • BullMQ gives you robust background jobs for provider syncing
  • Easy to hire for and find help online

Watch out for

  • Single-threaded — CPU-heavy analytics need careful handling
  • TypeScript types are optional, not enforced at runtime

Python with FastAPI — a modern, async-first Python framework that auto-generates API documentation from your type hints. Python is the natural home for data-heavy work: Pandas for analytics computation, NumPy for statistical calculations (luck index, consistency ratings), and first-class LLM library support. Celery handles background jobs for provider syncing. The auto-generated Swagger docs are a nice bonus for debugging your Swift networking layer. The trade-off is that Python is slower than Node for raw API throughput, though for this app the bottleneck will be database queries, not the server.

Works well when

  • Best ecosystem for data analytics (Pandas, NumPy, SciPy)
  • First-class LLM/AI library support
  • Auto-generated API docs from type hints

Watch out for

  • Slower raw throughput than Node or Go
  • Celery is powerful but complex to configure
  • Async Python has more gotchas than Node's event loop

Auth, database, realtime, storage — all managed
Let Supabase handle the infrastructure. Supabase gives you a hosted PostgreSQL database, built-in auth, realtime subscriptions, file storage, and Edge Functions (serverless Deno/TypeScript) — all out of the box. There's an official Swift SDK, so your SwiftUI app talks directly to Supabase. Complex analytics can run as database functions (PL/pgSQL) or Edge Functions. This gets you to a working product fastest since auth, database, and realtime are pre-built. The downside is you're locked into Supabase's patterns, and long-running provider sync jobs don't fit cleanly into Edge Functions (they have time limits).

Works well when

  • Fastest path to working product — auth, DB, realtime built-in
  • Official Swift SDK for direct integration
  • Realtime subscriptions for live score updates

Watch out for

  • Edge Functions have time limits — bad for long provider syncs
  • Complex analytics may outgrow database functions
  • Vendor lock-in — harder to migrate away later

Swift everywhere — shared models between app & API
Go all-Swift: use Vapor, the most mature server-side Swift framework, for the backend. Since your iOS app is already in Swift, you can share data models, validation logic, and types between the app and server using a Swift Package. Vapor has built-in support for job queues, WebSockets, and an ORM (Fluent). The appeal is one language everywhere — no context-switching between Swift and TypeScript/Python. The trade-off is a much smaller ecosystem: fewer libraries for provider API integrations, weaker LLM SDK support (you'd call Claude/OpenAI via raw HTTP), and a smaller community to get help from.

Works well when

  • One language (Swift) for everything — shared models and logic
  • Excellent performance — compiled, strongly typed
  • No context-switching between languages

Watch out for

  • Much smaller ecosystem — fewer libraries for integrations
  • No official LLM SDKs — raw HTTP calls to Claude/OpenAI
  • Smaller community — harder to find answers and examples

Conversation History Storage

Technical

When a user asks "who was my best manager in 2023?" and you answer, does that context carry forward to the next question? Where do those messages live — in Postgres, in Redis, on the device, or nowhere at all? This decision determines whether chat feels like a persistent memory or a disposable notepad, and how much the API needs to manage state between requests.

Option A
Recommended Chosen

Persistent DB Threads

model ChatThread { id String @id @default(cuid()) userId String leagueId String createdAt DateTime @default(now()) messages ChatMessage[] } model ChatMessage { id String @id @default(cuid()) threadId String role String // "user" | "assistant" content String sentAt DateTime @default(now()) thread ChatThread @relation(...) }
Store every message in two Prisma tables: ChatThread (one per league per user) and ChatMessage (every turn). When a new message comes in, the API loads the last N messages from that thread and includes them in the GPT-4o request as conversation history. Chat feels continuous — you can close the app, come back a week later, and the AI still knows what you were talking about. This is how ChatGPT works.

Works well when

  • You want multi-turn conversation that feels natural
  • Users ask follow-up questions ("what about the year before?")
  • History browsing is a future feature you want to leave open

Watch out for

  • Long threads get expensive — need a turn limit (last 10–20 turns)
  • Two new Prisma tables and a migration

Express API
stateless — never stores messages

The iOS app holds the full conversation in memory (and optionally persists it locally). Each API request includes the entire message history in the POST body. The Express API is completely stateless — it just passes history to GPT-4o and streams back a reply. Simple to build on the server side, but the API payload grows with every message. If the user deletes the app or switches devices, all history is gone.

Works well when

  • Simplest possible API — no DB changes needed
  • Single device, single user, low message volume

Watch out for

  • History lost if app is deleted or user reinstalls
  • Large payloads — 20-turn history sent on every request
  • No server-side visibility into conversations

Option C
Chosen

Redis Session Cache

iOS App
↓ sends threadId + new message
Express API
↕ read/write recent turns
Redis (TTL 24h)
no permanent storage
Use Redis (which you already run) to cache active conversation history with a time-to-live of 24 hours. The API reads recent turns from Redis, calls GPT-4o, appends the new exchange, and writes it back. After 24 hours of inactivity the conversation evaporates automatically. You get multi-turn memory within a session without adding Postgres tables, but nothing persists long-term.

Works well when

  • You want session-scoped memory without a DB migration
  • Redis is already in the stack (it is)

Watch out for

  • History evaporates after 24h — users lose context unexpectedly
  • Redis is not designed for durable storage — data can be lost on restart

Stateless — No Memory

iOS App
↓ sends one message at a time
Express API
↓ system prompt + single user message
GPT-4o
every question is independent
Every question is completely independent. The API sends only the system prompt, the league context, and the single user message — no prior conversation history. It's the simplest possible implementation, but it means the model has no memory at all. "What about the previous season?" makes no sense to it. This is more of a Q&A interface than a real chat.

Works well when

  • Absolute simplest to build — no state management at all
  • Questions are always self-contained (rare in practice)

Watch out for

  • No follow-up questions — model can't reference prior answers
  • Feels broken compared to any chat product users have used

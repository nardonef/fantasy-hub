Decision 13 of 19

Ingestion Data Model

Technical

Right now your Prisma schema has no concept of a "Player" -- players are just name strings on DraftPick and Transaction records. The Intelligence Platform needs to store signals from Reddit, Twitter, NFL stats, and consensus rankings, all tied to specific players. This decision is about how to structure those new tables: do you keep it simple with one big signals table, or normalize into separate tables per source type? The choice affects how easy it is to query across sources, how migrations work as you add new source types, and how the Player Resolution step (Phase 2) connects raw mentions to canonical player identities.

Exactly what you described in your build plan: one Signal table plus a new Player table. Every ingestion job writes to the same table, differentiated by source and sourceType enums. Source-specific data (like Reddit upvotes, tweet likes, or ranking position) goes into a flexible metadata JSON column. This is the simplest possible schema -- two new tables, one migration, done.

Works well when

  • You want to ship ingestion fast with minimal schema churn
  • Cross-source queries are common ("all signals for player X")
  • You're still figuring out what metadata matters per source

Watch out for

  • JSON metadata column is untyped -- no schema validation at DB level
  • As signal volume grows, one table could get very large

Instead of one generic table, create separate tables for each signal type: SocialMention (Reddit + Twitter), PlayerRanking (FantasyPros, ESPN, Yahoo), PlayerStat (NFL weekly stats), and PlayerNameAlias (for fuzzy match resolution). Each table has typed columns specific to its domain -- no JSON metadata bag. All link back to the Player table via FK. This is my recommendation because it gives you type safety, better query performance (index on rank, week, etc.), and makes each ingestion job's schema explicit. The cost is more tables and slightly more migration work upfront.

Works well when

  • You know the shape of each data source (you do -- the plan is detailed)
  • You want indexed queries like "rankings for player X, week 5, PPR format"
  • Type safety matters -- Prisma generates typed models per table

Watch out for

  • More tables = more migration steps when adding a new source type
  • Cross-source "unified feed" query requires JOINs or UNION

A middle ground: keep one Signal table for shared columns (source, player, content, sentiment, week), but add "extension" tables (RankingDetail, StatDetail) that link 1:1 to a Signal row when extra typed data is needed. Social mentions (Reddit, Twitter) just use the base Signal row. Rankings and stats get their own typed extension. This gives you a unified feed query on the base table, plus typed access when you drill into specifics.

Works well when

  • You want a unified feed query AND typed access for specific sources
  • New social sources (YouTube, TikTok) can be added with zero migration
  • The base table stays lean -- extensions are only joined when needed

Watch out for

  • 1:1 extension pattern adds JOIN complexity for detail queries
  • Prisma doesn't have native "table inheritance" -- you'll manage the pattern manually

Store everything as raw events in a single append-only table with a jsonb payload column. Then create PostgreSQL materialized views that project the data into queryable shapes -- one view for the feed, one for rankings aggregation, one for player stats. The raw event log is your source of truth; the views are rebuilt on a schedule (or triggered after each ingestion run). This is the most flexible approach if you expect the data shape to evolve a lot, but it adds operational complexity with view refresh management.

Works well when

  • Data shape is highly uncertain and you want to iterate without migrations
  • You want to replay or reprocess raw data (e.g., when LLM enrichment arrives)
  • Materialized views give you fast reads without denormalizing in app code

Watch out for

  • Materialized view refresh adds operational complexity and latency
  • JSONB queries are slower than typed columns for filtering/indexing
  • Prisma has limited support for materialized views -- you'll write raw SQL

Platform & Frontend Stack (iOS)

Technical

We're building an iOS app for Fantasy League Hub. The big question: do you go fully native with Swift, or use React Native to write in JavaScript/TypeScript? Native gives you the best possible iOS experience — buttery animations, Swift Charts, deep system integration. React Native lets you share language with your backend (TypeScript everywhere) and iterate faster, with the option to add Android or web later. For a data-heavy app with charts, tables, and an AI chat, both paths are viable — the trade-offs are around polish vs. velocity and how much you value feeling "truly iOS."

Go all-in on Apple's modern framework. SwiftUI gives you declarative UI that feels instantly "iOS" — native navigation, gestures, and animations out of the box. The killer feature here is Swift Charts, Apple's own charting framework — it's gorgeous, performant, and deeply integrated with accessibility and Dynamic Type. For a data-heavy analytics app, having charts that feel like a native part of iOS is a real differentiator. You also get seamless access to APNs (push notifications), widgets, Live Activities for game-day scores, and SharePlay for watching stats with leaguemates. This is my recommendation because the analytics experience will feel premium and the iOS-specific features (widgets, Live Activities) are perfect for fantasy football.

Works well when

  • You want the most polished, "feels like Apple made it" experience
  • Swift Charts gives you beautiful, accessible data viz for free
  • iOS-only features matter: widgets, Live Activities, SharePlay

Watch out for

  • Swift is a different language from your backend — no code sharing
  • SwiftUI still has rough edges with complex lists and navigation
  • If you ever want Android, you're starting from scratch

Build the iOS app with React Native using Expo's managed workflow. You write TypeScript — the same language as your backend — so you can share type definitions, validation logic, and data models between the app and server. Expo handles the build pipeline, over-the-air updates (push a fix without going through App Store review), and gives you a solid set of native modules out of the box. Charting is good but not great — Victory Native and react-native-chart-kit work, but they're not as polished as Swift Charts. The big win is development speed and the option to add Android later with the same codebase.

Works well when

  • TypeScript everywhere — shared types and logic with backend
  • Faster development cycle with hot reload and OTA updates
  • Android becomes possible later without a rewrite

Watch out for

  • Charts and data tables won't feel as native or polished
  • No access to iOS-specific features like Live Activities or widgets
  • Performance can lag on complex animated charts

Use the right tool for each screen
Use SwiftUI for most of the app — dashboards, the AI chat interface, settings, onboarding — but drop into UIKit for screens that need it, like complex data tables, custom collection layouts, or highly customized scroll behavior. This is how many production iOS apps work in practice. SwiftUI is great for 80% of screens, but fantasy analytics will have dense data grids (matchup history, roster comparisons) where UIKit's UICollectionView gives you more control. You still get Swift Charts, widgets, and all the native goodies.

Works well when

  • You want native polish but need UIKit's power for dense data views
  • Battle-tested approach used by most large iOS apps
  • Best of both worlds — SwiftUI speed + UIKit flexibility

Watch out for

  • Two UI paradigms to maintain — more cognitive overhead
  • Navigation between SwiftUI and UIKit screens can be tricky
  • Still no code sharing with backend

Get the best of both worlds: write most of the app in React Native (TypeScript, fast iteration, shared logic with backend) but build custom native modules in Swift for the parts that matter most — Swift Charts for gorgeous data viz, native widgets for home screen stats, and Live Activities for game-day scores. This is the "have your cake and eat it too" option. The trade-off is complexity: you're maintaining two languages, a bridge layer, and the bare React Native workflow (no Expo managed convenience). It's more work to set up but gives you TypeScript velocity where it helps and native quality where it counts.

Works well when

  • You want TypeScript velocity AND native iOS features
  • Swift Charts via native module for premium data viz
  • Can still share types and logic with backend

Watch out for

  • Most complex setup — two languages, bridge layer to maintain
  • No Expo managed workflow — harder build pipeline
  • Native module bugs are harder to debug than pure Swift or pure RN

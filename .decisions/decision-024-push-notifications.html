Decision 24 of 25

Push Notification Infrastructure

Technical

Push notifications span the full stack: iOS needs to register for notifications and handle deep links from taps, the Express API needs to store device tokens and dispatch notifications, and something needs to connect your server to Apple's Push Notification service (APNs). The choice is between building directly on APNs (full control, no third party), using Firebase Cloud Messaging as a middle layer, or a managed service that handles the delivery complexity. Since social is low priority, this only needs to cover AI-related events: weekly recap ready, and optionally new AI insights.

Use the node-apn (or apn) npm package to send directly to Apple's Push Notification service from your Express API. You store device tokens in a new device_tokens Prisma table, and a BullMQ job dispatches notifications when events fire. No third-party dependency in the delivery path — just your server talking to Apple. Requires an APNs key (.p8 file) from your Apple Developer account, which takes about 5 minutes to generate.

Works well when

  • No third-party dependency — just your server and Apple
  • iOS-only app, no Android to worry about
  • Full control over retry logic and delivery tracking
  • No ongoing cost beyond your server

Watch out for

  • Must manage APNs certificate/key rotation yourself
  • No delivery analytics out of the box

Use Firebase Cloud Messaging as a middle layer — your Express API sends to FCM via firebase-admin, and FCM handles the APNs delivery. The benefit: FCM adds delivery receipts, handles token refresh automatically, and is Android-ready if you ever expand. The cost: a Google/Firebase project dependency for an iOS-only app, and an extra hop in the delivery chain. FCM's free tier is generous and more than enough for a personal app.

Works well when

  • Delivery receipts and analytics matter
  • Future Android expansion is possible
  • Auto token refresh reduces maintenance

Watch out for

  • Google dependency for an iOS-only app adds unnecessary complexity
  • Extra delivery hop adds latency
  • Firebase project setup + config files in iOS app

OneSignal is a managed push notification service — you add their SDK to the iOS app, make REST API calls from Express, and they handle all APNs complexity. Comes with a dashboard, delivery analytics, segmentation, and A/B testing you'll never use. Free tier covers everything you need. The downside: an SDK in your iOS app and a third-party in the critical delivery path. Overkill for a personal app sending weekly recaps.

Works well when

  • Fastest to implement — dashboard handles APNs config
  • Rich delivery analytics and segmentation built in
  • Free tier is extremely generous

Watch out for

  • Third-party SDK in iOS app for a simple use case
  • Overkill — you don't need segmentation or A/B for this app
  • Vendor dependency for a core user-facing feature

Option D
Chosen

Defer — In-App Notifications Only

Instead of push, show on next app open:
🏆
Fantasy Hub — New
Your Week 7 recap is ready
You went 1-1 this week. Frank's team scored a league-high 142pts…
Shown as a banner/badge on Chat tab — no APNs needed
Don't implement push notifications for V2 at all. When a weekly recap is ready, surface it as an in-app notification badge on the Chat tab — users see it the next time they open the app. This is zero infrastructure work: no APNs certificates, no device token table, no BullMQ notification job, no iOS permission prompt. Push notifications can be added in V3 once the recap content is validated and users actually want the interruption.

Works well when

  • Social is low priority (already decided) — push fits social, not AI chat
  • Zero infrastructure overhead — ship AI chat faster
  • Validate recap content before interrupting users' phones

Watch out for

  • Users must open the app to see recaps — no re-engagement loop
  • Push infra needs to be built eventually anyway

Decision 23 of 25

AI Chat Tool Inventory

Technical

The tool inventory defines exactly what the AI can look up. Each tool maps to a Prisma query that runs against your league data. The Jahnel principles are clear: type your schemas, handle null results explicitly, and design parallel vs. sequential call patterns deliberately. This decision also determines whether tools call the existing V1 analytics endpoints (reuse) or query Prisma directly (more control). The inventory directly maps to progress.md sub-tasks.

Thin Wrappers Over V1 Endpoints

Calls existing V1 route
pgvector semantic search
get_standings(year?)→ /analytics/standings — season records, ranks, PF/PA
get_h2h(manager_a, manager_b?, year?)→ /analytics/h2h — head-to-head records between managers
get_scoring(year?)→ /analytics/scoring — avg, max, min PPG per manager
get_draft(year)→ /analytics/draft — picks, grades, round/manager grid
get_records()→ /analytics/records — champions, top scores, nail biters
get_playoffs(year?)→ /analytics/playoffs — clutch ratings, playoff PPG
get_manager_profile(manager_id)→ /analytics/dashboard — career stats, rivals, rank history
search_league_history(query)→ pgvector semantic search across all embeddings
Each AI tool is a thin wrapper that calls the existing V1 analytics Express routes internally — same Prisma queries, same logic, no duplication. Tools call route handler functions directly (not HTTP), so there's no self-referencing HTTP overhead. The search_league_history tool adds pgvector semantic search on top. This maximally reuses tested V1 code and means fixing a bug in the standings query fixes it for both the iOS analytics view and the AI chat.

Works well when

  • Reuses all 8 tested V1 analytics queries — no duplication
  • Bug fixes in V1 routes automatically benefit AI tools
  • Tool inventory maps 1:1 to existing endpoint test coverage

Watch out for

  • V1 route responses shaped for iOS, not necessarily ideal for LLM context
  • Tool can't do cross-query joins (e.g. standings + h2h in one call)

Direct Prisma, AI-Shaped Responses

Direct Prisma query
pgvector semantic search
query_standings(year?, limit?)→ Prisma: returns compact {name, wins, losses, pf, rank}
query_h2h(manager_name_a, manager_name_b)→ Prisma: all-time record + recent 5 matchups
query_scoring_trends(manager_name?, year?)→ Prisma: weekly scores array for trend analysis
query_draft_history(manager_name?, year?)→ Prisma: picks with position, round, value context
query_champions()→ Prisma: champion per year with margin-of-victory
query_manager_stats(manager_name)→ Prisma: career overview, playoff record, rivals
query_transactions(manager_name?, type?, year?)→ Prisma: trades, adds, drops with context
search_league_history(query)→ pgvector: semantic search across embeddings
Tools query Prisma directly and return responses shaped for LLM consumption — compact, denormalized, with names instead of IDs. For example, query_h2h returns {"frank vs blake: 23W-18L-0T, last 5: WWLWL"} — the kind of compact summary an LLM can use without further processing. Accepts manager names (not IDs) since users will say "frank" not a UUID. Slightly more work to build but produces better AI answers.

Works well when

  • Response shape is optimized for LLM context, not iOS rendering
  • Tools accept natural names ("frank") not internal IDs
  • Cross-cutting queries possible (one tool, multiple relations)

Watch out for

  • Duplicates some logic from V1 routes — two places to update on schema changes
  • More code to write and test than Option A

Minimal Set (3 Tools)

Broad SQL query
pgvector
query_stats(category, filters)→ One mega-tool: category = standings|h2h|scoring|draft|records|playoffs
get_manager_info(manager_name)→ All career stats for a specific manager in one call
search_history(query)→ pgvector semantic search — fallback for everything else
Collapse all stats retrieval into one query_stats(category, filters) mega-tool plus a get_manager_info tool. Simple to define, simple to maintain. The downside: a mega-tool with a category enum is harder for the LLM to use correctly — it has to know the right category to pass. Fine-grained tools are easier for GPT-4o to reason about than one tool with many configurations.

Works well when

  • Fastest to implement — 3 tools instead of 8
  • Minimal surface area to maintain

Watch out for

  • Mega-tools are harder for LLMs to call correctly than specific tools
  • Can't do parallel tool calls (LLM can't call standings + h2h simultaneously)
  • Error handling is coarser — one tool failing blocks everything

Option B + Smart Router Tool

Router (cheap model)
Direct Prisma
classify_query(question) → tool_names[]→ GPT-4o-mini decides which tools to call before main model runs
query_standings / query_h2h / …→ Same 7 direct Prisma tools as Option B
search_league_history(query)→ pgvector semantic search
Same tool set as Option B, but add a lightweight classify_query step first: GPT-4o-mini reads the question and returns the list of tools that should be called, then the full model runs with that pre-selected tool set. This is the Jahnel "small model routing" principle in action — cheaper classification before expensive generation. Reduces token cost per message at scale. Adds ~200ms latency and extra complexity.

Works well when

  • High query volume makes per-message cost meaningful
  • Router prevents main model from calling irrelevant tools
  • Applies Jahnel small-model routing principle directly

Watch out for

  • GPT-4o is already good at tool selection — router may not add much
  • Extra latency + extra API call per message
  • Premature optimization for a personal app with low query volume

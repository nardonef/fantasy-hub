Decision 5 of 12

Auth & Multi-Tenancy

Technical

Fantasy League Hub is a multi-user platform — people sign up, connect their leagues, and invite leaguemates to share analytics. That means we need user authentication, but also a permissions model: who can see which league's data? How do provider OAuth tokens (especially Yahoo's) get stored and refreshed? The tricky part is that a single user might be in multiple leagues across different providers, and a single league has multiple members who all need access. We also need to store each user's Yahoo/ESPN credentials securely so background sync jobs can fetch on their behalf.

Clerk handles sign-up/login; you handle league access & provider tokens
Use Clerk (a managed auth service) for user sign-up, login, and session management — it handles email/password, Apple Sign-In, Google, and magic links out of the box with a native iOS SDK. Then build your own league permissions layer on top: a league_members table in Postgres that maps users to leagues with roles (owner, member, viewer). Provider OAuth tokens (Yahoo, ESPN) are stored encrypted in a separate provider_tokens table that only the sync workers access. I recommend this because Clerk eliminates weeks of auth work (password reset flows, session management, iOS keychain integration) while keeping your league permissions flexible and under your control.

Works well when

  • Auth is handled — sign-up, login, sessions, Apple Sign-In all built-in
  • Native iOS SDK with keychain integration
  • League permissions stay flexible and under your control

Watch out for

  • Monthly cost scales with active users ($25/mo + per-user after 10k)
  • Provider OAuth (Yahoo) still needs custom handling

Build auth yourself using Passport.js (the standard Node.js auth library) with JWT tokens. Passport has strategies for Apple Sign-In, Google, email/password, and crucially — Yahoo OAuth, which means your auth flow and provider connection flow can be the same thing. When a user "connects Yahoo," they're both authenticating and granting API access in one flow. JWTs are stored in the iOS keychain. You own everything: user table, session management, token refresh logic. No third-party costs or dependencies. The trade-off is building all the auth plumbing yourself — password reset, email verification, rate limiting, and security hardening.

Works well when

  • Zero external auth costs — free at any scale
  • Yahoo OAuth as both auth and API access in one flow
  • Full control over every aspect of the auth experience

Watch out for

  • Weeks of work to build properly — password reset, email verify, rate limits
  • Security is your responsibility — auth bugs are high-stakes
  • Session management and token refresh are easy to get wrong

Firebase handles identity; your API handles everything else
Use Firebase Authentication for user identity (it's free up to 10k monthly active users, then very cheap), with your Express API verifying Firebase ID tokens on each request. Firebase has an excellent iOS SDK with Apple Sign-In, Google, email/password, and phone auth built in. Your Express backend verifies the Firebase token, looks up the user in your Postgres database, and handles league permissions and provider tokens. Firebase Auth is one of the most battle-tested auth services — it powers millions of apps. The limitation is that provider OAuth (Yahoo) is separate from Firebase auth, so connecting a Yahoo league is a different flow from signing up.

Works well when

  • Free up to 10k MAU — most generous free tier
  • Excellent iOS SDK — Apple Sign-In, Google, phone auth
  • Battle-tested at massive scale (Google infrastructure)

Watch out for

  • Google dependency — some users/devs prefer to avoid
  • Provider OAuth (Yahoo) is completely separate from Firebase auth
  • Less customizable than Clerk for the sign-up/login UI

Use Supabase just for auth — keep your own Postgres for everything else
Use Supabase's auth module (built on GoTrue, an open-source auth server) without using the rest of Supabase. It has a Swift SDK, supports Apple Sign-In, email/password, magic links, and phone auth. Free for up to 50k monthly active users. Your Express API verifies Supabase JWTs and handles league logic against your own PostgreSQL instance. The advantage over Firebase is that Supabase Auth is open-source — you could self-host it later if you wanted to. The downside is that the Swift SDK is less mature than Firebase's iOS SDK.

Works well when

  • Most generous free tier — 50k MAU free
  • Open-source (GoTrue) — can self-host later if needed
  • Swift SDK with Apple Sign-In support

Watch out for

  • Swift SDK is newer and less polished than Firebase's
  • Smaller community — fewer iOS-specific examples
  • Still need custom Yahoo OAuth handling

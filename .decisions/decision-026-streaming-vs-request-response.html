Streaming vs. Request / Response

Technical

GPT-4o can start sending words back before it's finished thinking — like watching someone type in real time. Or it can think silently and then send the whole answer at once. This affects how the Express API is written, how the iOS app handles the response, and — most importantly — how the chat feels to use. A complex fantasy question with tool calls can take 5–8 seconds. What happens during that wait matters a lot.

Option A
Recommended Chosen

SSE Token Streaming

Who was my best manager last season?
Based on your 2024 season, Frank led all managers with 1,847 total pts and a .714 win rate across 14 week
tokens arrive continuously — first word in ~800ms
The Express API sets Content-Type: text/event-stream and pipes OpenAI's streaming response directly to the client. On iOS, URLSession reads the response byte-by-byte using AsyncBytes, appending each token to the message as it arrives — the same typewriter effect you see in ChatGPT. The user sees the first word almost immediately, even for long answers. When tools are called mid-response, you can show a "looking up your stats…" indicator while the tool runs.

Works well when

  • Answers are long — streaming fills the wait naturally
  • Tool calls happen mid-response (show progress indicators)
  • This is what users expect from modern chat AI

Watch out for

  • More complex iOS code — need to handle chunked byte stream
  • Saving the final message to DB requires accumulating the full stream

Wait for Full Response

Who was my best manager last season?
silence for 5–8s, then full answer appears
Standard POST request — Express calls OpenAI with stream: false, waits for the complete response, then returns it as JSON. iOS gets the whole answer at once and displays it immediately. Much simpler to implement on both sides. The downside is that for a question requiring tool calls (e.g., "look up my 2023 draft grades and compare to final standings"), the user stares at a spinner for 5–8 seconds with no indication of progress.

Works well when

  • Simplest implementation — standard JSON POST/response
  • No complex byte-stream handling on iOS

Watch out for

  • 5–8s blank spinner is a bad experience for tool-heavy queries
  • iOS URLSession may timeout on slow GPT responses

Option C
Chosen

Stream + Phase Events

1 User sends message
↓
2 SSE event: phase: "thinking"
↓
3 SSE event: phase: "tool_call" → "Fetching your standings…"
↓
4 SSE tokens: answer streams in
An extension of Option A — the SSE stream sends both token chunks and structured phase events. When GPT-4o decides to call a tool, the server emits a data: {"type":"tool_call","name":"getStandings"} event before the tool runs, which iOS can use to show a contextual indicator like "Looking up your standings…". This is the most polished UX — the user always knows what's happening. It's also the most complex to implement.

Works well when

  • Queries regularly involve multiple tool calls
  • You want a premium, transparent UX

Watch out for

  • Most complex of all options — custom event protocol on top of SSE
  • iOS needs to parse both token chunks and structured events in the same stream

WebSocket Connection

ws://api/chat
← OPEN persistent connection
→ send: { message: "..." }
← recv: { token: "Based" }
← recv: { token: " on" }
← recv: { token: " your..." }
← recv: { done: true }
bidirectional — connection stays open
Use WebSockets (via ws npm package) to maintain a persistent connection between iOS and Express. The client sends messages as JSON frames and receives token frames back in real time. WebSockets are designed for bidirectional real-time communication — but this is a one-direction streaming problem. You're adding significant infrastructure complexity (connection lifecycle, reconnection logic, iOS URLSessionWebSocketTask) for no benefit over SSE, which was built exactly for this use case.

Works well when

  • You need server-to-client AND client-to-server real-time events
  • Multiple simultaneous live updates (not this use case)

Watch out for

  • Overkill — SSE is simpler and purpose-built for one-way streaming
  • Connection lifecycle management adds significant complexity
